# 缺货统计系统 v3.21.0 综合优化执行计划

> 版本：v3.21.0 | 日期：2026-05-25 | 状态：规划中
> 项目：WSZH-ShortageStore | 综合两轮深度分析，去重合并 40 项

---

## 🟢 第一批：高优先级（成本低、收益大，建议 1 周内完成）

| 编号 | 分类 | 优化项 | 现存问题 | 具体方案 | 预期效果 |
|:--:|------|--------|----------|----------|----------|
| **P1-01** | 安全 | **sync_data.py 硬编码数据库密码** | 第22-28行直接写了 SQL Server 密码和 Supabase service_role key，提交到 Git 即所有人可见 | 改为 `os.environ.get()` 读环境变量，创建 `.env.example` 模板 | 消除敏感信息泄露风险 |
| **P1-02** | 稳定性 | **26 处空 catch 块加日志** | store.js(11处)、admin.js(4处)、ESF(9处)、scheduled-task(2处) 使用 `catch(e){}` 空捕获，问题无法排查 | 全部改为 `catch(e){console.error('[模块] 操作失败:', e);}`，admin.js 额外写入 sync_log | 问题可追溯，排查效率×10 |
| **P1-03** | 数据库 | **SPFXB_Result 查询加 NOLOCK** | `get_store_inventory`、`get_product_detail`、`get_purchase_plan` 读 SPFXB_Result 无 NOLOCK，SPFXB 刷新期间可能锁等待 | SQL 语句加 `FROM dbo.SPFXB_Result WITH (NOLOCK)` | 刷新库存与汇总查看不再互相阻塞 |
| **P1-04** | 数据库 | **shortage_storestock_cache 加索引** | 按 store_name 或 product_code 查询时无独立索引，全表扫描 | `CREATE INDEX ON shortage_storestock_cache(store_name)` + `(product_code)` | 缓存查询 200ms → 5-10ms |
| **P1-05** | 安全 | **越权访问拦截加强** | `get_my_reports` 从前端传 store_id，用户可篡改查看别店数据 | 后端从 JWT sub 解析身份，不从 params 读 store_id | 杜绝水平越权 |
| **P1-06** | 体验 | **缺货汇总表加排序** | 无法按"总需求""订货数量"排序 | 表头加点击排序（纯前端，基于 filteredData） | 管理员分析效率提升 |
| **P1-07** | 体验 | **密码重置加二次确认** | 点「重置密码」无确认，误操作后员工无法登录 | `showConfirm('确定重置该员工密码为默认密码？')` | 防止误操作 |
| **P1-08** | 代码 | **sync_data.py 服务化** | sync_data.py 需手动执行或 --daemon，崩溃后不重启 | Windows Task Scheduler 每15分钟触发，脚本加 PID 文件锁防重复 | 同步全自动化零人工 |
| **P1-09** | 响应 | **get_all_products 去 pinyin_code** | 向前端返回 5000 条含 pinyin_code，前端只用前 4 个字段 | ESF 中只返回 product_code/name/spec/manufacturer | 传输减少 ~20% |
| **P1-10** | 响应 | **get_store_inventory 精简字段** | 预加载返回 16 个字段，首页只展示 6 个 | 预加载只返回核心 6 字段，详情查询返回全量 | 传输减少 ~40% |

| **P3-08** | 部署 | **Electron 版本号统一** | package.json/electron-main.js/check-update 三处硬编码 | electron-main 用 `app.getVersion()`，ESF 读环境变量 | 发布少改 2 处 |
